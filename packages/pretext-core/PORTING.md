# pretext-core porting contract

C++ port of the text engine mugen runs on React Native today:
`@chenglou/pretext@0.0.8` (layout/line-break kernel) + `@wingleeio/pretext-native`
(font-table advance ruler + JS segmenter). The port targets **exactly the
semantics that execute on Hermes today** ŌĆö not the browser paths.

## Non-negotiable rules

1. **Byte-identical output.** Same doubles (bit-for-bit), same break points,
   same line texts as the TS engine running on Node with pretext-native's
   polyfills force-installed. The conformance harness (`test/conformance.test.ts`)
   compares with `Object.is` per number. To achieve this:
   - Every JS `number` is a C++ `double`. Never `float`. Integer-valued
     bookkeeping (indices, counts) may be `int32_t`/`size_t` when the TS value
     is provably integral.
   - Port arithmetic **in the same order** as the TS source. Do not refactor
     `a + b + c` into `a + (b + c)`, do not hoist, do not "simplify" math.
   - No `-ffast-math`; CMake sets `-ffp-contract=off`.
2. **Strings are UTF-16.** Use `std::u16string` / `std::u16string_view`
   everywhere in the kernel. All indices are UTF-16 code-unit indices, exactly
   like JS `string.length` / `.slice()` / `.charCodeAt()`. `codePointAt`
   semantics: unpaired surrogates yield the surrogate code unit itself.
3. **Mirror the TS file structure 1:1.** One C++ module per TS module, same
   function names (camelCase preserved), same internal helper names, same
   comments where they explain semantics. A reviewer must be able to diff the
   C++ against the TS side-by-side.
4. **Native-only simplifications (these are the ONLY allowed deviations):**
   - `EngineProfile` is a compile-time constant: `{ lineFitEpsilon: 0.005,
     carryCJKAfterClosingQuote: false, breakKeepAllAfterPunctuation: true,
     preferPrefixWidthsForBreakableRuns: false, preferEarlySoftHyphenBreak: false }`
     (`getEngineProfile()` on RN: `navigator.userAgent === ''` ŌåÆ not Safari,
     not Chromium; same as the `typeof navigator === 'undefined'` branch).
   - `emojiCorrection ŌēĪ 0` (requires DOM; RN never has one). Consequently
     `getCorrectedSegmentWidth(seg, m, 0) == m.width`, `textMayContainEmoji`,
     `getEmojiCorrection`, `countEmojiGraphemes` need not be ported. KEEP the
     function `getCorrectedSegmentWidth` as a passthrough so call sites mirror
     the TS.
   - `getMeasureContext()` / canvas: `ctx.measureText(seg).width` is replaced
     by a direct call to `fonts::measureTextWidth(seg, font)` ŌĆö that is exactly
     what pretext-native's `OffscreenCanvasShim` does.
   - `Intl.Segmenter` is replaced by the C++ port of pretext-native's
     `segmenter/{grapheme,word}.ts` (NOT ICU ŌĆö hermetic, matches Hermes today).
   - `setLocale`/locale is ignored (pretext-native's segmenter ignores it).
5. **Module-level caches stay module-level.** TS module globals (e.g.
   `segmentMetricCaches`, `resolvedCache`, `parseCache`) become
   function-local `static` state in the matching `.cpp`, with the same
   `clearŌĆ”()` functions exported. Single-threaded access is assumed (JSI calls
   arrive on the JS thread; the fixture runner is single-threaded).
6. **Unicode property regexes** become lookups in `cpp/tables/unicode_tables.hpp`
   (generated by `scripts/gen-unicode-tables.mjs` from the exact regexes on
   Node ŌĆö do not hand-copy ranges). If your module needs a class that is not
   yet in the generator, ADD it to the generator script and re-run
   `node scripts/gen-unicode-tables.mjs`.
   Literal character sets written out in TS (e.g. `kinsokuEnd`, quote sets)
   are ported inline as explicit `switch`/set code, keeping the TS order/comments.
7. **JS container semantics.** `Map` iteration order or key semantics never
   affect output in this codebase (caches only). Arrays ŌåÆ `std::vector`.
   `Set<string>` of single chars ŌåÆ fast switch on char16_t/char32_t.
   JS `string` keyed caches ŌåÆ `std::unordered_map<std::u16string, ŌĆ”>` (font
   shorthand keys stay UTF-16 or UTF-8 ŌĆö pick one per module and be consistent;
   font shorthands are ASCII in practice but MUST NOT crash on non-ASCII).
8. **Errors**: TS `throw new Error(msg)` ŌåÆ `throw std::runtime_error(utf8msg)`.

## Prepared handles

`PreparedText` / `PreparedTextWithSegments` are plain structs
(`pretext::Prepared`), owned by the caller via `std::shared_ptr<Prepared>`.
`Prepared.segments` is only populated when prepared with segments (mirrors
`includeSegments`). The line-text cache (`line-text.ts` `getLineTextCache`)
lives ON the Prepared struct (`mutable` member) instead of a WeakMap.

## Callbacks

`walkPreparedLinesRaw` / `walkLineRanges` / `walkRichInlineLineRanges` take
callbacks; port them as template-free `const std::function<ŌĆ”>&` (matching the
TS callback arguments exactly).

## Conformance

`tools/fixture-runner` builds a host CLI: reads a JSON request stream
(one op per line ŌĆö register fonts, prepare, layout, layoutWithLines,
naturalWidth, rich-inline ops), writes JSON results with numbers serialized
via shortest-round-trip (`std::to_chars`, then formatted to match JS
`String(x)`: see tools/fixture-runner/number_format.hpp). The vitest harness
parses both sides and compares with `Object.is` per number and `===` per
string, so serialization only needs round-trip fidelity, not textual equality.
